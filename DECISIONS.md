# DECISIONS — judgment calls made autonomously

1. **Prototype path**: mission says `./prototype/` but the actual folder is `./mohammed-haggag-courses_UI/` — used as-is, not renamed (it is the design reference, untouched).
2. **Workspace layout**: 3 independent pnpm projects (`/api`, `/web`, `/admin`) + a root `package.json` with convenience scripts (`pnpm seed`, `pnpm dev`) that delegate into the projects — keeps the "exactly 3 sibling projects" rule while giving the one-command DX the DoD asks for.
3. **Framework versions**: used what the official CLIs scaffold today (NestJS 11, Next.js 15, Tailwind v4) rather than pinning older majors — CLAUDE.md mentioned Nest 10/Next 14 but the mission says to use the official CLI; newest stable wins.
4. **Styling strategy**: ported the prototype's inline-style + CSS-custom-properties approach verbatim (tokens in `globals.css`) instead of rewriting every style as Tailwind classes — guarantees pixel fidelity and matches the prototype's code style; Tailwind is installed and used for layout utilities where convenient.
5. **DB modeling**: Category embeds its `groups[]`/`levels[]` (small, bounded, edited together in one admin modal). Lessons are a separate collection keyed by `categoryId + groupId? + levelId` with an `order` field — they're unbounded and reordered independently.
6. **Admin credentials**: kept the prototype's `admin / admin123` as the seeded admin (User with `username: "admin"`, role `admin`). Students log in by email; the admin login endpoint accepts username or email.
7. **Auth sessions**: single active refresh token per user (hash stored on the user doc, rotated on refresh). Multi-device sessions are out of scope for MVP.
8. **Seeded lessons**: replicated the prototype's deterministic `genLessons()` (5–8 lessons per level, 12–40 min, titles from the 8 templates, first lesson free, demo YouTube id `aqz-KE-bpKQ`) so seeded content matches the prototype exactly.
9. **FAQ / why-cards / learn list / access steps / contact / terms** had no admin editor in the prototype → modeled them inside the SiteContent singleton and ADDED editors for all of them in the admin "المحتوى العام" tab (mission rule: every content area must be editable).
10. **Lesson gating shape**: public endpoints never include `youtubeId`. A dedicated `GET /lessons/:id/watch` returns it only for free lessons, unlocked users, or admins. The e2e test asserts the id never appears anywhere in public payloads.
11. **Checkout flow**: `POST /orders` (pending) → provider adapter `createPayment` → `POST /orders/:id/pay` (mock confirm) → order `paid` + unlock. Mirrors a real PSP redirect/webhook shape while staying mock. Both `paymob` and `stripe` adapters implement one `PaymentProvider` interface and always succeed.
12. **Admin-created students**: the add-student modal requires the admin to set an initial password (prototype had no password concept). Plan column in the students table is derived from the user's latest paid order.
13. **e2e tests** run against a real MongoDB (the docker instance) using a separate `haggag_e2e` database that is dropped per run — avoids the mongodb-memory-server binary download and tests the real driver path.
14. **Continue-watching** is stored server-side on the user (last 8, deduped by lesson), matching the prototype's localStorage behavior but cross-device.
15. **Currency on checkout**: order stores `amount + currency` snapshotted from the plan at purchase time (prices may be edited later in admin; historical orders keep what was actually charged).
16. **Next.js version**: `create-next-app@latest` installed `16.3.0-preview.0` (npm `latest` points at a preview). Pinned both fronts to **16.2.8**, the newest stable line. Next 16 notes honored: `params`/`searchParams` are Promises; `middleware.ts` is removed → no proxy file at all; route protection is client-side redirects + real enforcement in the API guards.
17. **No fake sample data**: the prototype seeded demo students/payments for the admin tables. The real admin shows only real users/orders — seeding fakes would pollute production-looking data. Empty states are implemented instead.
18. **Non-native track level names**: prototype reused UAE school-grade names («الصف الأول»…) for the Arabic-for-non-natives track although its description says «ثلاثة عشر مستوى». Improved to «المستوى الأول…الثالث عشر» with the same note («مستوى تواصلي»).
19. **Contact form** is real: `POST /contact` stores messages in Mongo (`contactmessages`); `GET /contact` (admin) exists for retrieval. No admin UI tab for it (out of scope) — view via mongo-express.
20. **Admin "إعادة الضبط" (reset) button dropped**: it reset the prototype's localStorage DB; with a real database a one-click destructive reset is a footgun. Re-seeding is `pnpm seed`.
21. **Section headings** that are pure layout copy (e.g. «لماذا هذا الأستاذ؟», eyebrows, button labels) stay static; every substantive content block (hero, instructor, why-cards, learn list + learn-section copy, access steps, FAQ, testimonials, plans, contact info, terms, footer blurb, categories/lessons) is DB-backed and admin-editable.
22. **Web page architecture**: each route = thin server `page.tsx` (metadata + `serverApi` fetch, `cache: 'no-store'` so admin edits show on refresh) + one client component for interactivity. Admin panel pages are fully client-side (cookie-authed fetches) behind a client guard layout.
23. **Lesson player poster**: the prototype showed the YouTube thumbnail behind the play button. Locked/free ids never reach the catalog payloads, so the pre-play poster is the navy gradient + ornament; the thumbnail would leak the id of locked lessons via the page source otherwise.
24. **Docker Mongo moved to host port 27018**: this machine runs a homebrew `mongod` on 127.0.0.1:27017 (used by other projects). With docker also publishing 27017 (on `*`), Node's connection pool raced IPv4/IPv6 for `localhost` and ended up with sockets to BOTH servers — queries nondeterministically hit a full or an empty database. The stack now binds docker Mongo to `127.0.0.1:27018` (explicit IP, unambiguous). The stray `haggag`/`haggag_e2e` databases this created on the host mongod were dropped; no other host databases were touched.
25. **Mongoose 9 regression — ref-path string casting**: `find({ category: '<24-hex string>' })` matches nothing in Mongoose 9 (only `_id` paths still auto-cast). All filters on ref paths (`category`, `user`) now cast explicitly via `new Types.ObjectId(...)`. A regression test was added to the e2e suite (admin lesson listing, reorder, `/orders/mine`).
26. **Mongoose 9 subdocument spread**: `{ ...plan.prices }` yields subdocument internals, not values (broke the plan-prices PATCH with required-field validation errors). Nested updates merge field-by-field instead.
27. **Anonymous `/auth/me` returns `200 {user:null}`** instead of 401 — a 401 printed a red console error on every page for every visitor. Clients attempt one silent refresh only when a localStorage session hint exists (cookies live on the API origin, so a cookie hint can't be read cross-origin).
28. **Production env validation fails fast**: with `NODE_ENV=production`, missing `MONGO_URI`/JWT secrets/origins or weak secrets (<32 chars, containing `change-me`/`secret`) abort boot. Dev keeps friendly defaults.
29. **Rate limiting only on credential endpoints** (register/login/admin-login 10/min, refresh 30/min per IP; `/me`+`/logout` exempt) — a global throttle would break SSR, because the web server's single IP proxies *all* visitors' content fetches.
30. **Helmet + `/health` + graceful shutdown + `TRUST_PROXY`/`SWAGGER_ENABLED`/`COOKIE_SAMESITE` env switches** added for deployment; Swagger should be disabled in production.
31. **Seed admin credentials are env-overridable** (`ADMIN_USERNAME`/`ADMIN_EMAIL`/`ADMIN_PASSWORD`); the seed warns loudly when the default `admin123` is used.
32. **Deployment packaging**: per-app multi-stage Dockerfiles (Next `output: standalone`), `docker-compose.prod.yml` (Mongo not published to the host; only the three apps exposed), `.env.prod.example` with strong-secret placeholders. `NEXT_PUBLIC_*` values are baked at build time → they are docker **build args**; the web server-side `API_URL` stays runtime-configurable.
33. **Bilingual public web (Arabic default + English), UI-only scope**: the web app translates UI chrome + all API exception/validation messages, but DB-stored **content** (category/level names, lesson titles, hero, why, FAQ, plans, testimonials, terms) stays Arabic. No data-model change, and the admin panel stays Arabic-only. Rationale: lessons are Arabic-teaching content; full content i18n would mean `{ar,en}` on every field plus EN admin editors — far larger and out of the requested scope. (Scope chosen explicitly by the user.)
34. **Locale transport = cookie + `Accept-Language`, no URL `[locale]` segment**: the `mh_locale` cookie is the single source of truth — read server-side via `cookies()` in the root layout & `generateMetadata`, client-side via `LocaleProvider`. The header AR/EN toggle writes the cookie, imperatively flips `<html lang/dir/class>`, and calls `router.refresh()`. No locale route segment (avoids restructuring all routes + reintroducing middleware). Because the API is a different origin, the web cookie isn't sent to it; the browser client forwards locale as the `Accept-Language` header on every request instead.
35. **API i18n = message-key catalog + global exception filter, Arabic default**: every user-facing exception/validation message is a short key (`MK.*`) defined once in `api/src/i18n/messages.ts` as `{ar,en}`. A global `I18nExceptionFilter` resolves the request locale (`?lang` → `Accept-Language` → cookie → default `ar`) and swaps the key for localized text, preserving the standard `{statusCode,message,error}` shape. Unknown strings pass through untranslated (graceful/incremental). Content endpoints are untouched. AR text is byte-identical to the literals it replaced, so the e2e suite stays green. (Category enum values, the admin `وصول كامل` data label, and the Swagger description remain Arabic — they are content, not messages.)
36. **Web dictionary split per-namespace** (`lib/i18n/{ar,en}/<area>.ts`, composed in `index.ts`; `Dict = typeof ar`, English enforced at compile time via `satisfies Record<Locale, Dict>`): one file per page-area enables conflict-free parallel editing and gives a compile-time key-parity guarantee (missing/renamed EN keys fail `tsc`). **Latin font**: Inter added via `next/font`; `html.loc-en { --font: var(--font-inter) }` swaps only the sans stack — `--serif` stays Amiri so Arabic accents (brand glyph, the footer Quran ayah) still render correctly in English mode. Numbers are locale-aware via `fmtNum` / `useLocale().num` (Eastern-Arabic digits for `ar`, Western for `en`).
37. **CapRover deploys + root-context Dockerfiles (single source of truth)**: CapRover's build context is always the repository root and it needs a captain-definition file. Rather than keeping a second CapRover-specific set of Dockerfiles (guaranteed drift), the three existing Dockerfiles were rewritten to root context (`COPY api/ .` etc.), `docker-compose.prod.yml` now builds with `context: . / dockerfile: <app>/Dockerfile`, and one root `.dockerignore` replaced the three per-app ones (only the context root's `.dockerignore` applies). Root captain-definitions: `captain-definition` (default → api, so the existing api app config works unchanged) plus `captain-definition-api`/`-web`/`-admin` for explicit per-app paths set in the dashboard's DEPLOYMENT tab. `NEXT_PUBLIC_*` build args flow from CapRover App Config env vars (CapRover passes app env vars as build args; the Dockerfiles declare matching `ARG`s). All three images were built root-context locally and smoke-tested (api `/health` + data, web 200, admin 200) before pushing.
